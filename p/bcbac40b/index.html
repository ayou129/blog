实战：腾讯实时语音视频通话

创建于：2020年3月5日上次更新：2022年12月7日

功能背景

公司产品是一款社交性APP，用户群体大部分是年轻人(18-30岁为主)。

所以诞生了优先级很高的一个产品需求：语音、视频通话功能，整体上讲，也能提高产品整体营收。

任务

与前端Android和IOS同事确认事实方案后，完成产品开发以及测试。

行动

由于该功能是基于腾讯IM的，所以我阅读了腾讯云相关的功能介绍，并与同事进行了沟通，明确了各自分工、共同制定通话场景的业务文档(心跳、开始和结束通话)、工作优先级、日期规划(完成、提测、正式上线日期)等

个人难点：

  1. 通话功能中嵌套了计费功能
  2. 语音通话、视频通话存在相同的特殊场景(网络异常的含义是指心跳无法按照原约定定时请求服务器)，如下

特殊异常场景

影响因素：

  • 多用户
  • 断网情况(网络异常、进程异常)

正常通话以及挂断场景

用户A 与 用户B、用户C 的各种场景

  1. A 请求与 B 通话(正常场景)
  2. B 未接通前 A 挂断(正常场景)
  3. B 未接通前 B 挂断(正常场景)
  4. B 接收后 A 挂断(正常场景)
  5. B 接收后 B 挂断(正常场景)

进程异常场景(例如：在手机中将 app shuwdown)

  1. A 请求与 B 通话(正常场景)
  2. B 未接通前 A 杀死进程(非正常场景)
  3. B 未接通前 B 杀死进程(非正常场景)
  4. B 未接通前 A、B 同时杀死进程(非正常场景)
  5. B 接收后 A 杀死进程(非正常场景)
  6. B 接收后 B 杀死进程(非正常场景)
  7. B 接收后 A、B同时杀死进程(非正常场景)

网络异常场景

  1. A 请求与 B 通话(正常场景)
  2. B 未接通前 A 网络异常(非正常场景)
  3. B 未接通前 B 网络异常(非正常场景)
  4. B 未接通前 A、B 同时网络异常(非正常场景)
  5. B 接收后 A 网络异常(非正常场景)
  6. B 接收后 B 网络异常(非正常场景)
  7. B 接收后 A、B同时网络异常(非正常场景)

网络异常场景-重新恢复

  1. A 请求与 B 通话(正常场景)
  2. B 未接通前 A 网络异常，一段时间后恢复网络，继续通话(非正常场景)
  3. B 未接通前 B 网络异常，一段时间后恢复网络，继续通话(非正常场景)
  4. B 未接通前 A、B 同时网络异常，一段时间后恢复网络，继续通话(非正常场景)
  5. B 接收后 A 网络异常，一段时间后恢复网络，继续通话(非正常场景)
  6. B 接收后 B 网络异常，一段时间后恢复网络，继续通话(非正常场景)
  7. B 接收后 A、B同时网络异常，一段时间后恢复网络，继续通话(非正常场景)

代码实施

服务端API讲解（此API是客户端请求的，非腾讯服务器请求）

重点补充

  • 每个通话都有一个唯一Key和房间号，字段设计的方案为 产品名+通话类型+IM房间号+唯一key+用户A(u_ID)+用户B(u_ID)。
  • 通话时长、用户所剩时长由后端redis key进行统计，因为需求中有：用户所剩时长还剩xx秒的时候要提示。
  • 每个通话有多条心跳Log，进行统计和分析。
  • 定时脚本处理 脱机的语音通话: 使用Redis scan处理异常的通话房间，并且上报，为了追溯且定位原因。
    定时脚本：trtc分支的TrtcController.php

总结

  1. 考虑到并发的问题，所以使用redis+lua脚本的方案解决货币计算等问题，思路和实现都OK。
  2. 和同事沟通成本最大的地方在梳理通话时的不同场景，很多种异常情况都要考虑。
  3. 完成这个功能之后，对自己的成长收益特别高，同时通话也是自己最满意的一个功能之一。
  4. 完成语音及视频通话功能且上线稳定后，月统计出平均日收益提升至23个百分点，老大还给了奖金，开心！
