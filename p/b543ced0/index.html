RabbitMQ

创建于：2019年4月3日上次更新：2022年10月2日

RabbitMQ基础知识+生产环境所遇到的问题的总结。

RabbitMQ 是什么？

采用 AMQP(高级消息队列协议) 的一种消息队列技术。

AMQP的作用

  • 解耦
    • 例子：用户注册后有一系列操作： 头像剪裁压缩水印、汇总该用户可能认识的人、将用户的整个一体系信息汇总到可视化表中
      通过消息队列分配给 具有不同单一职责的消费者 进行处理，极大程度上解耦。
  • 异步应用
    • 例子：一个请求中使用异步处理可以 缩短请求时间，并且可以 避免一个请求中任何环节出现问题 影响业务逻辑。
  • 流量削峰
    • 一天内每个时段系统请求的数量都不同，存在 请求高峰期，对于mq来说，设置合理的消费者每s消费的数量，在不影响系统稳定性的前提下依然能够处理数据。

AMQP的缺陷

  • 降低系统可用性
    • 引入外部依赖越多，系统稳定性越低，MQ挂了应用就停滞数据处理，业务造成影响。
  • 系统复杂度提高
    • 会出现许多问题，如何保证消息没有重复消费？如何处理消息丢失的情况？如何保证消息传递的顺序性？
  • 一致性问题
    • 使用异步消息队列分成若干个小系统，有的系统会处理失败，对用户体验造成影响。

如何选择(各个AMQP产品的对比)

AMQP 单机吞吐量 topic数量对吞吐量的影响 时效性 可用性 消息可靠性 功能支持
ActiveMQ w级 - 毫秒级 高可用，主从架构 有较低的概率丢失数据 MQ领域中的功能及其完备
RabbitMQ w级 - 微妙级，低延迟 高可用，主从架构 - 基于erlang开发，并发能力很强，性能好，延时很低
RockerMQ 10w级 topic达到几百、几千的级别，吞吐量会有较小幅度的下降，这是RocketMQ的优势，在同等机器的数量下，可以支持大量的topic 毫秒级 非常高，分布式架构 参数优化后可以0丢失 MQ功能完善，支持分布式扩招
Kafka 10w级 topic达到几十到几百个的时候，吞吐量会大幅度下降，在同等的机器数量下，Kafka尽量保证topic的数量不要过多，如果需要支持支撑大规模的topic，需要增加机器资源 延迟在毫秒级以内 非常高，分布式架构，一个数据多个副本，少数机器宕机，不会丢失数据 参数优化后可以0丢失 功能较为简单，支持简单的MQ功能，主要用于大数据领域，如实时计算以及日志采集等场景，缺点就是学习成本高，比如需要额外学习zookeeper(分布式协调服务)

总结：

  • ActiveMQ 最早使用，但是没经历过大吞吐场景验证，社区不活跃
  • RabbitMQ 稳定，高吞吐，适合单机或主从部署
  • RocketMQ 稳定，阿里出品，进入了Apache基金会，社区活跃，支持分布式
  • Kafka 稳定，社区活跃，适合大数据领域的实时计算、日志采集等场景，支持分布式

RabbitMQ 概念

  • Broker：简单来说就是消息队列服务器实体。

  • Exchange：消息交换机，它指定消息按什么规则，路由到哪个队列。

  • Queue：消息队列载体，每个消息都会被投入到一个或多个队列。

  • Binding：绑定，它的作用就是把exchange和queue按照路由规则绑定起来。

  • Routing Key：路由关键字，exchange根据这个关键字进行消息投递。

  • vhost：虚拟主机，一个broker里可以开设多个vhost，用作不同用户的权限分离。

  • producer：消息生产者，就是投递消息的程序。

  • consumer：消息消费者，就是接受消息的程序。

  • Connection：脚本程序 和 amqp程序 之间真实的 TCP 连接。

  • Channel：虚拟通道，复用 Connection节省系统开销，可建立多个channel，每个channel代表一个会话任务。

    在生产中： 一个 消费者 一个 channel(出于 监听queue 和 消费速度 考虑)，对于 生产者 复用一个 channel 就行。

工作流程

  1. php脚本(客户端)连接到消息队列服务器(tcp进程)，并打开一个channel(线程)。
  2. php脚本(客户端)声明一个 exchange、routing key、queue，绑定好其关系并设置相关属性。
  3. php脚本(客户端)投递消息到exchange。
    • exchange 接收到消息后，就根据消息的 key 和已经设置的 binding，进行消息路由，将消息投递到一个或多个队列里。
    • exchange 也有几个类型，完全根据key进行投递的叫做 Direct 交换机，例如，绑定时设置了 routing key 为”abc”，那么客户端提交的消息，只有设置了key为”abc”的才会投递到队列。对key进行模式匹配后进行投递的叫做 Topic 交换机，符号”#”匹配一个或多个词，符号”*”匹配正好一个词。例如”abc.#”匹配”abc.def.ghi”，”abc.*”只匹配”abc.def”。还有一种不需要key的，叫做 Fanout 交换机，它采取广播模式，一个消息进来时，投递到与该交换机绑定的所有队列。

持久化

三者都需要开启持久化，缺一不可

  1. 交换器持久化
  2. 队列持久化
  3. Message持久化

投递失败

什么情况下产生的？

  • 生产者将消息 传递到RabbitMQ后PHP故障没有回应，MQ拒收

解决方案：

  • 消息失败后记录信息，后台可视化界面判断是否重复发送

重复消息

什么情况下产生的？

  • 生产者重复：生产者将消息 传递给到RabbitMQ后，RabbitMQ故障，迟迟没有给生产者确认，生产者可能重复发送。

  • 消费者重复：处理完数据后 RabbitMQ宕机或其他(网络波动，内部错误，超时) 没有及时返回ack

    详细解释：
    在互联网应用中，尤其在网络不稳定的情况下，消息队列RabbitMQ版的消息有可能会出现重复。如果消息重复消费会影响您的业务处理，请对消息做幂等处理。消息重复的可能原因如下：

    • 发送时消息重复

      当一条消息已被成功发送到服务端并完成持久化，此时出现了网络闪断或者客户端宕机，导致服务端对客户端应答失败。 如果此时Producer意识到消息发送失败并尝试再次发送消息，Consumer后续会收到两条内容相同并且Message ID也相同的消息。

    • 投递时消息重复

      消息消费的场景下，消息已投递到Consumer并完成业务处理，当客户端给服务端反馈应答的时候网络闪断。为了保证消息至少被消费一次，消息队列RabbitMQ版的服务端将在网络恢复后再次尝试投递之前已被处理过的消息，Consumer后续会收到两条内容相同并且Message ID也相同的消息。

    • 负载均衡时消息重复（包括但不限于网络抖动、服务端重启以及Consumer应用重启）

      当消息队列RabbitMQ版的服务端或客户端重启、扩容或缩容时，会触发Rebalance，此时Consumer可能会收到重复消息。

解决思路：使用幂等性概念解决

幂等性: 执行一次和多次产生的影响均相同

解决方法：

  1. 在数据库中创建一个 含有唯一索引的key 在消费时开启事务并 Insert 唯一Message ID
  2. 解决重复发送：发送的时候ack确认，生成唯一ID(查询表验证唯一性) 并传递 唯一Message ID
  3. 解决重复消费：消费的时候ack确认，并且以唯一 Message ID 判断是否重复消费(Mysql事务)，对于重复投递的消息：记录+不执行+确认ACK，等待管理员判断

发送者确认模式

  1. 无法路由，使用 mandatory = ture
    • 进而使用备用交换器(声明交换器时，将备用捆绑到主交换器上)，可以将无法路由的 Message 提交，避免无法路由的情况
  2. 可以路由，但是内部异常，使用发送者通知模式 confirmSelect，并且可以使用异步模式
    • 成功返回 hack
    • 失败返回

消费者确认模式

  1. get 手动确认

  2. consume 自动确认
    可以返回 reject 拒绝该 Message，拒绝后，进入队列中重新分发，可以声明并进入到 死信队列

    死信队列(声明队列时)：1.消息过期 2.队列过期 3.队列消息数满了，后续的进入死信队列

  3. ack 单条数据执行成功，返回已执行标识

  4. nack 单条数据执行失败，返回未执行标识

死信Exchange

https://help.aliyun.com/document_detail/148081.htm?spm=a2c4g.11186623.0.0.44e61cf4zhvLbh#concept-2341972

死信Exchange适用于处理被消费者否定应答或重试失败的消息。

消息变成死信消息的可能原因如下：

  • requeue参数被设置为true，消息重试超过16次，消息重试失败
  • requeue参数被设置为false，消费者使用basic.reject或basic.nack否定应答（NACK）消息。
注意 过期消息或超过队列最大长度被丢弃的消息不会成为死信消息。

路由流程：

  1. 消费者消费失败后消息重试次数超过16次依然没有成功，或者消费者没有消费消息而直接否定应答，消息变成死信消息。
  2. Queue根据x-dead-letter-exchange将死信消息发送到死信Exchange，并根据x-dead-letter-routing-key为死信消息设置死信Routing Key。
  3. 死信Exchange将死信消息路由到死信Queue。

延时队列

  • 简介:带有延迟功能的消息队列

  • 用处:

    1. 订单未支付，自动更新为 过期 状态
    2. 支付相关的分频通知请求

发布消息时的权衡

事务

  1. 有严重的性能问题(降低2-10倍的性能)
  2. 使生产者应用产生同步

解决方案：

  1. 开启失败通知 mandatory ，解决无法路由的问题
  2. 开启发送方确认模式，交换器将信件通过路由交给队列时，通知生产者投递成功
    • 揽收成功 ack
    • 发生内部错误无法投递进入队列 nack

注意事项

  1. 1线程 对应 1信道，n线程 共享 1TCP连接
  2. 消息需要确认，如果一直不确认，则会一直存在
  3. 多个消费者共同绑定多个交换器，MQ会轮询消费者
  4. 生产者发送message，没有路由键匹配和相应的配置(mandatory=true打开发消息时的失败通知)，则会被丢弃
    无法获取消息是否发送成功 的状态
  5. 如果exchange和queue都是持久化的，那么它们之间的binding也是持久化的。如果exchange和queue两者之间有一个持久化，一个非持久化，就不允许建立绑定。
  6. 当路由键找不到主交换器时(无法路由)，进入备用交换器进行消费
